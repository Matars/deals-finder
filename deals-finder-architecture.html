deals finder — actual data flow

traces exactly what happens from cron trigger to published page. no database exists — everything lives in LLM context.

How the LLM "scrapes"

  • • LLM navigates browser to each search URL
  • • Injects site-specific JavaScript via browser_console()
  • • JS returns raw JSON arrays (title, url, price)
  • • Each site has different DOM — different JS per source
  • • Blocket: <article> elements
  • • Tradera: aria-describedby price lookup
  • • Vinted: URL-slug title extraction

⚠ Why This Breaks

  • • ~45 browser navigations per run (9 cats × 5 sources)
  • • Each = 3-5 tool calls = 135-225 total tool calls
  • • All scraped data sits in LLM context (~100K tokens)
  • • Cron sessions have limited tool-call budget
  • • 2026-04-19: ran out of iterations before git push
  • • No persistent state between runs — re-scrapes everything daily
  • • original_price is estimated (LLM guesses), not from site

The "Database" Problem

  • • There is no database. Zero. None.
  • • Data: site → LLM context → write_file → git
  • • No way to query past deals, track prices over time,
  • or deduplicate against yesterday's scrape
  • • SQLite would fix this:
  • - Store all deals with timestamps
  • - Track price changes per item
  • - Deduplicate without LLM memory
  • - Generate JSON from DB (not from LLM)

Skill-Based Architecture

  • • blocket-second-hand-search — DOM extraction JS, URL patterns, gotchas per site
  • • github-repo-management — git clone/push workflow
  • • Skills are just markdown files injected into the LLM prompt
  • • LLM reads skill, follows instructions, uses tools
  • • No code execution — LLM IS the scraper + processor
  • • cron-github-pages-deals-publisher — the meta-skill that defines the whole workflow pattern

Data Flow Summary

  • 1. ⏰ Cron fires → spawns fresh LLM session
  • 2. 📥 LLM gets: prompt + 2 skills + tool access
  • 3. 🌐 LLM navigates browser → scrapes 5 sites
  • 4. 🧠 LLM processes in memory (no DB)
  • 5. 📝 LLM writes JSON + HTML files
  • 6. 🔀 git add → commit → push to GitHub
  • 7. 🌍 Pages serves static site
  • 8. 💬 Discord: 1 compact link message

To Make This "Real"

  • • Add SQLite — persistent deal storage, price tracking
  • • Separate scraper from processor — Python script does extraction, LLM just categorizes
  • • Split cron into per-source jobs (parallel)
  • • Cache search results — don't re-scrape unchanged categories
  • • Real original_price from retailer APIs (not LLM estimates)
  • • Contact-seller workflow on the static site
